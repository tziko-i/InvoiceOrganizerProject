הצעת פרויקט

שם הפרויקט: InvoiceOrganizer

תיאור תמציתי:
המערכת מהווה פלטפורמה אינטרנטית מקיפה לניהול, ארגון וניתוח של חשבוניות וקבלות עבור יחידים ועסקים קטנים. היא מאפשרת למשתמשים להעלות חשבוניות, לסווג אותן לקטגוריות ולספקים, ולצפות בניתוח ויזואלי של הוצאות לאורך זמן. המערכת כוללת לוח בקרה (Dashboard) דינמי, ניהול משתמשים בטוח, ויכולת ייצוא דוחות מתקדמים.

מטרות עסקיות ויעדים:
לייעל את המעקב הפיננסי באמצעות צמצום הצורך בתיוק ידני ויצירת מאגר דיגיטלי מאובטח ונגיש מכל מקום. מתן כלים להבנת 'לאן הולך הכסף' באמצעות ניתוחים סטטיסטיים ויזואליים.

טכנולוגיות מרכזיות:
צד לקוח מבוסס Angular 21 עם ממשק PrimeNG, צד שרת מבוסס ASP.NET Core 9 (Web API), ומסד נתונים Entity Framework Core מעל SQLite.

1.3 תכולת הפרויקט

1.3.1 תאור ורקע כללי

ניהול פיננסי תקין מחייב מעקב רציף אחר הוצאות. קבלות וחשבוניות נצברות בקצב מהיר, לרוב בצורה לא מסודרת (נייר, מיילים, תמונות בטלפון). הפיזור הזה מקשה על עריכת תקציב, הגשת דוחות מס ואיתור כשלים פיננסיים. פרויקט InvoiceOrganizer בא לתת מענה טכנולוגי מתקדם לבעיה זו על ידי הפיכת הניירת הקשיחה והמפוזרת למסד נתונים דיגיטלי מרכזי הניתן לניתוח מהיר ונוח.

1.3.2 מטרות המערכת

  • ריכוז מידע: יצירת מאגר מרכזי, בטוח ומאורגן לכל מסמכי הרכישה.
  • מניעת אובדן נתונים: הבטחת נגישות למסמכי הוכחת קנייה (אחריות חומרה, החזרות) לאורך זמן.
  • ניתוח עסקי ופרטי (BI): סיפוק כלים ויזואליים להבנה פשוטה של דפוסי ההוצאה החודשיים והשנתיים חלוקה לקטגוריות.
  • נגישות: בניית מערכת מבוססת ענן המאפשרת גישה, העלאה ועריכה מכל דפדפן ומכשיר נייד בצורת (Responsive UI).

1.3.3 סקירת מצב קיים בשוק, אלו בעיות קיימות

כיום, הגישות הרווחות בשוק מתחלקות לכמה סוגים:

  • תיוק קלסרים מסורתי: גישה מיושנת הדורשת זמן רב, מקום אחסון, חשופה לבלייה ואינה מאפשרת חיפוש.
  • שמירה בכונן/ענן (Google Drive/Dropbox): הניירת אמנם דיגיטלית, אך עדיין חסרה היכולת לשאוב את הנתונים לצורך ציפייה בגרפים ומעקב תקציבי אוטומטי.
  • מערכות עסקיות כבדות (ERP/חשבשבת/פריוריטי): מערכות מעולות אך יקרות, מסורבלות מאוד למשתמש הביתי או לעסק הזעיר הלא-מיומן.

1.3.4 מה הפרויקט אמור לחדש או לשפר

המערכת מציעה פתרון הוליסטי ופשוט ללקוח הקצה. היא מחברת בין עולם האחסון לעולם האנליטיקה. פרויקט InvoiceOrganizer משפר דרמטית את חווית המשתמש (UX) עיצוב Modern Bright, מספק לוחות בקרה מהירים המקבלים נתונים חיים (Real-Time charts), ומשלב טכנולוגיות צד-שרת חזקות (.NET 9) ועדכניות בצורה כזו שגם המשתמש הכי פשוט יוכל לנהל את הוצאותיו כאילו מדובר בחברה עסקית מתקדמת.

1.3.5 דרישות מערכת ופונקציונליות

1.3.5.1 דרישות מערכת

סביבת דפדפן: תמיכה בדפדפנים מודרניים (Chrome, Edge, Safari, Firefox).
שרידות וביצועים: המערכת צריכה להחזיר תשובה תוך שניות בודדות גם בטעינת מאות רשומות, תוך שימוש במנגנוני Lazy Loading ו-Pagination בצד לקוח (Angular Table) ובחיתוכים בסיס מסד נתונים.
התמודדות עם עומסים: תשתית השרת ASP.NET Core מיועדת לעבד בקשות מרובות במקביל (Asynchronous Controller Actions) כולל עיבוד קבצים במבני Multipart/form-data.

1.3.5.2 דרישות פונקציונליות

  • מודול משתמשים: אפשרות להרשמה (Registration), התחברות (Login) וניהול Session מבוסס Token.
  • מודול מסמכים: העלאת תמונות ו-PDF של חשבוניות. אפשרות להזנת פרטי החשבונית: מס', תאריך, ספק, סכום.
  • לוח בקרה אינטראקטיבי: הצגת גרף מגמות חודשי, תרשים פיזור הוצאות לקטגוריות ורשימת ספקים מרכזיים.
  • רשימות תומכות: ניהול עצמאי של רשימת קטגוריות (למשל: דלק, מזון, ציוד משרדי) וניהול רשימת ספקים לצורך שיוך מהיר בחשבונית חדשה.

1.3.6 בעיות צפויות במהלך הפיתוח ופתרונות

1.3.6.1 תיאור הבעיות

  • בעיית אינטגרציה מעגלית במידע: בשימוש ב-Entity Framework, אובייקטים המקושרים תלותית (למשל, חשבונית שמכילה פריטים שהם בתורם מקושרים חזרה לחשבונית) יוצרים Reference Cycles בהחזרת JSON, מה שגורם לקריסת הבקשה.
  • תאימות צד-לקוח למסכים: תצוגת טבלאות נתונים ארוכות וגרפים במכשירי סלולר נוטה "להישבר" ולחרוג מרוחב המסך.
  • בעיות תמיכה בעברית מול שרתות בסיס נתונים: התמודדות עם תאריכים מיון וכיוון כתיבה עברי (RTL) ביישום.

1.3.6.2 פתרונות אפשריים

  • פתרון מעגליות: הוספת הקונפיגורציה ReferenceHandler.IgnoreCycles בהגדרות השירותים (Program.cs) למניעת קריסה.
  • פתרון תאימות רספונסיבית: שימוש במערכת הרשת המובנית (Grid System) של PrimeNG לצד שאילתות CSS (@media queries) שמתאימות את הסרגל האנכי לתצורת תפריט נייד.
  • פתרון תאימות נתוני זמנים ושפה: קביעת מבני נתונים (DateOnly / DateTimeUtc) במקום סתם מחרוזות, ואכיפת סידור מיון (OrderByDescending) בשרת ולא בלקוח.

1.3.7 פתרון טכנולוגי נבחר

1.3.7.1 טופולוגית הפתרון

המערכת בנויה בארכיטקטורת שרת-לקוח (Client-Server Architecture) קלאסית. צד הלקוח (Frontend) רץ בדפדפן של המשתמש ומתקשר עם צד השרת (Backend) באמצעות קריאות HTTP מאובטחות (REST API) על גבי פרוטוקול TCP/IP. השרת אחראי לניהול הלוגיקה העסקית, התקשורת מול בסיס הנתונים (Database) המקומי, והתקשורת ל-APIs חיצוניים.

1.3.7.2 טכנולוגיות בשימוש

בפרויקט שולבו טכנולוגיות מודרניות ליצירת אפליקציה מהירה ואמינה (SPA - Single Page Application):
Frontend: Angular בגרסה 21 (המשתמשת ביכולות Standalone Components לטעינה מהירה), ספריית PrimeNG לניהול רכיבי הממשק (UI Components).
Backend: ASP.NET Core בגרסה 9.0 הפועל כ-Web API.
Database: SQLite כאמצעי קל ומהיר לאחסון נתונים (Relational DB) המיועד לפיתוח ופריסה פשוטה.

1.3.7.3 שפות הפיתוח

  • צד לקוח: HTML5, CSS3, TypeScript (השפה המרכזית ב-Angular).
  • צד שרת: C# (גרסה 12/13).
  • מסד נתונים: SQL (הלכה למעשה, מתורגם אוטומטית מקוד גישה מונחה-עצמים - Entity Framework LINQ).

1.3.7.4 תיאור הארכיטקטורה

ארכיטקטורת שכבות (N-Tier Architecture) בשרת:
שכבת התצוגה (Controllers): מקבלת את בקשות ה-HTTP, מבצעת Data Validation בסיסי (Model Binding).
שכבת הלוגיקה העסקית (Services): מבצעת את החישובים, עיבוד התמונות ויצירת הטוקנים (JWT).
שכבת הנתונים (Data Access / Repository): שימוש ב-Entity Framework Core לצורך מיפוי (ORM) וביצוע פעולות CRUD מול מסד הנתונים, מה שמנתק את רוב שורת הקוד מצורך בשאילתות SQL ארוכות.

1.3.7.5 חלוקה לתכניות ומודולים

מודול לקוח (Client App): מחולק לפי תכונות – Auth (התחברות ורישום), Dashboard (ניהול תקציב, גרפים), Invoices (העלאת קבצים, טבלאות נתונים), Reports (הורדת דוחות אקסל).
מודול שרת (API): מחולק ל-Controllers שונים: AuthController, InvoicesController, ReportsController, UsersController.

1.3.7.6 סביבת השרת( מקומי, וירטואלי, ענן, שירות אירוח)

סביבת השרת היא Hosting מקומי (Kestrel) בתוך סביבת Windows במהלך הפיתוח. לקראת ייצור, היישום תוכנן לפעול על שרת צד-שלישי תומך .NET Runtime (כגון Azure App Service או שרת וירטואלי מבוסס Linux בעזרת תהליך Reverse Proxy). הנתונים נשמרים מקומית בקובץ ".db" בנתיב השרת.

1.3.7.7 ממשק המשתמש/לקוח GUI

הממשק עבר עיצוב קפדני ומוקפד, מהסגנונות הכי עדכניים של "Modern Bright".
שימוש בספריית ה-UI של PrimeNG מאפשר רכיבים מוכנים כגון טבלאות עם מיון וסינון, גרפים (באמצעות Chart.js), כפתורים מדורגים, ואלמנטים קופצים (Modals) תוך חיסכון ניכר בזמן פיתוח ושמירה על קוד גמיש.

1.3.7.8 ממשקים למערכות אחרות API

נכון לגרסה הנוכחית, המערכת מתקשרת בעיקר כ-API עצמאי (First-Party API). בתוכניות ההרחבה שולבו יכולות התממשקות ל-Azure Document Intelligence (לצורך סריקת OCR אוטומטית מחשבוניות במייל) ומול דרישות צד שלישי לשליחת מיילים אוטומטיים (Resend).

1.3.7.9 שימוש בחבילות תוכנה

צד לקוח (Node Package Manager - NPM): rxjs, primeng, chart.js.
צד שרת (NuGet Packages): Microsoft.EntityFrameworkCore, System.IdentityModel.Tokens.Jwt, BCrypt.Net-Next (להצפנת סיסמאות), Microsoft.AspNetCore.Authentication.JwtBearer.

1.3.8 שימוש במבני נתונים וארגון קבצים

1.3.8.1 פרוט מבנה הנתונים

טבלת Users: מכילה את קוד המשתמש, אימייל, סיסמה מוצפנת (Hash), ותאריך הצטרפות.
טבלת Categories: לוח קטגוריות לסיווג הוצאה (מזון, דיור וכו').
טבלת Suppliers: רשימת הספקים, כתובותיהם ומזהה הקטגוריה השייך להם כברירת מחדל.
טבלת Invoices: טבלת הליבה האוגרת - מספר חשבונית (InvoiceNumber), תאריך, סך העסקה, נתיב פיזי של קובץ התמונה (FilePath), ומזהה הספק.
טבלת InvoiceItems: לניהול רמת הפריט בכל חשבונית – מזהה חשבונית, תיאור פריט, מחיר ליחידה, וכמות.

1.3.8.2 שיטת האחסון( מאגר, קבצים ובאיזה טכנולוגיה)

מאגר טבלאי: שימוש בבסיס נתונים יחסי מסוג SQLite המעניק גישה מהירה לשמירת שלמות המידע (Referential Integrity).
קבצים: תמונות וחשבוניות שמועלות (PDF/JPG) נשמרות פיזית בתיקיית Uploads בשרת. בסיס הנתונים מחזיק רק נתיב לקובץ (מניעת התנפחות מסד הנתונים).

1.3.8.3 מנגנוני התאוששות מנפילה\קריסה\תמיכה בטראנזקציות

המערכת מבססת Transactional Operations בעזרת ה-Entity Framework. במקרים בהם מוחדר מידע משולב (כגון חשבונית חדשה יחד עם שורות פריטים), במקרה של שגיאה באמצע התהליך מתבצע Rollback אוטומטי לשמירה על עקביות הנתונים.

1.3.9 תרשימי מערכת מרכזיים

1.3.9.1 Use Case

תרחיש מרכזי (Create Invoice):
הלקוח מתחבר למערכת -> מנווט לעמוד הוספת חשבונית -> הלקוח אקטורי מעלה קובץ ומזין סכום -> המערכת מעבדת את הנתונים -> המערכת שולחת אישור סטטוס ללקוח, ומציגה הודעת הצלחה.

1.3.9.2 Sequence diagram - רצף קריאות פונקציות מרכזיות

זרימת התקשורת בקריאת נתונים סטטיסטיים ללוח הבקרה:
[לקוח-אנגולר] -> קורא לפונקציית HttpClient.get -> [שרת API] Controllers מקבל בקשה -> מופעל מנגנון ה-TokenValidation -> קריאה ל-Service -> קריאה ל-DbContext ושאיבת מידע מסונן דרך LINQ -> החזרת אובייקט DTO דרך JSON -> ה-Frontend קולט את ה-JSON, ממפה אותו לסדרת נתונים (Arrays) ומזין ל-Chart.js להצגה.

1.3.9.3 Data flow

זרימת המידע מההזנה ועד הייצוג הוויזואלי:
קלט משתמש (טופס רישום חשבונית) --> הופך ל-Object JSON --> נשלח ב-HTTP POST לשרת --> מתורגם שם לאובייקט Invoice דרך ModelBinder --> נשמר לשורת נתונים יחסית ב-SQLite. בעת צורך בניתוח --> השרת אוסף חיתוכים מ-SQLite באמצעות שאילתות GroupBy --> מפיק רשומות Summary --> עובר ללקוח.

1.3.10 תיאור המרכיב האלגוריתמי חישובי

1.3.10.1 איזה בעיה בא לפתור, איך יפתור?

הבעיה: משתמשים רוצים לדעת מיד את דפוסי ההוצאה החודשיים, התפלגות הקטגוריות והספקי-קצה שמושכים את מרב ההוצאות, מבלי לחשב הכל ידנית כל פעם.
הפתרון: בקרת ה-Reports מחזיקה אלגוריתמיקה שמבצעת אגרגציה סטטיסטית. בשרת מתבצע תהליך חיתוך, איסוף וכימות אוטומטי של חשבוניות הלקוח על סמך זמנים. סכימה סלקטיבית זו חוסכת התחשבנויות מורכבות במחשב הלקוח.

1.3.10.2 איסוף מידע וניתוחים סטטיסטיים

המערכת אוספת בזמן אמת עשרות שורות נתונים גולמיים ממסד הנתונים. בעזרת הפונקציונליות GroupBy ו-Sum, מיוצר מיפוי שמציג באיזו קטגוריה ההוצאות חריגות, מה הממוצע החודשי מול הוצאה נוכחית, ועד כמה העסק\אדם קרוב לחריגת יעד.

1.3.11 תיאור/התייחסות לנושאי אבטחת מידע

המערכת מתמודדת עם מידע פיננסי רגיש הדורש רמת אבטחה טובה. מנגנוני ההגנה מבוססים על הטכנולוגיות הבאות:

  • הגנת זהות: סיסמאות משתמשים אינן נשמרות כטקסט גלוי אלא עוברות תהליך גיבוב קריפטוגרפי עמוק (Hashing + Salt) בעזרת חבילת BCrypt.
  • בקרת גישה למשאבים: מעבר בין שכבות דורש אימות זהות. השרת מנפיק Json Web Token (JWT) בעל תוקף מוגבל. כל בקשה לשרת נדרשת להציג טוקן תקין (Bearer Auth) המאומת ברמת ה-Middleware ב-ASP.NET Core. מנגנון זה מונע גישה ממי שאינו מורשה.
  • הגנה מפני התקפות Web: הספרייה עצמה (Angular) מגנה מפני הזרקות (XSS) על ידי Sanitization מובנה, והשרת מוגדר לדחות קריאות Cross-Origin חריגות (CORS Policy).

1.3.12 משאבים הנדרשים לפרויקט

1.3.14 תכנון הבדיקות שיבוצעו

1.3.14.1 בדיקות תהליכיות ברמת משתמש (Full Flow)

תרחיש בדיקה מרכזי: המשתמש רושם פרופיל חדש, מתחבר עם שם המשתמש, מקבל הרשאה ועובר לעמוד לוח הבקרה. הוא מנווט להוספת חשבונית ומעלה קובץ, המערכת מחזירה סטטוס "הועלה בהצלחה", הגרף התקציבי בלוח הבקרה קופץ חזותית בצורה מיידית לפי הסכום הנדרש.
תרחיש בדיקה חריג: הזנת סיסמה שגויה או טוקן שפג תוקפו יגרמו לדחיית הלקוח ולגירושו ישירות למסך ההתחברות תוך פליטת הודעת שגיאה חלקה שלא תגרום לקריסת הבק-אנד.

1.3.14.2 בדיקות יחידה (Unit Tests)

סדרת קריאות עצמאיות לבדיקת ה-API מנותק מסביבת ה-GUI (בד"כ ע"י Postman או פונקציות Mocha). למשל, שליחת בקשת GET לטיפול בראוט של הדוחות ללא הזנת Token מתאימה - צריכה להחזיר תגובה כירורגית HttpCode 401 Unauthorized.

1.3.15 בקרת גרסאות version control

הפרויקט מנוהל במערכת Git ומגובה לענן ב-GitHub. נשמרות גרסאות לפי נקודות ציון היסטוריות (Commits). מתבצע שימוש בענף Master ליציבות הפרויקט, וכאשר משולב פיצ'ר חדש כמו הרחבה של דו"ח Excel, כל התיקונים צפויים לחזור לענף הראשי מה שהופך את שחזור הקוד לבטוח בעת כל קריסה פיזית בכונן שרת העבודה.

2. הבטחת בטיחות: מחולל ומצפין תעודת אימות (TokenService.cs)

public string CreateToken(Users user) { // הוצאת מפתח ממסמך הקינפוג והמרתו לביטים var tokenKey = config["TokenKey"]; var key = new SymmetricSecurityKey(Encoding.UTF8.GetBytes(tokenKey)); // יצירת מטען Claims חיוניים var claims = new List<Claim> { new Claim(ClaimTypes.Email, user.Email), new Claim(ClaimTypes.NameIdentifier, user.Id) }; // הגדרת חתימה אלגוריתמית var creds = new SigningCredentials(key, SecurityAlgorithms.HmacSha512Signature); var tokenDescriptor = new SecurityTokenDescriptor { Subject = new ClaimsIdentity(claims), Expires = DateTime.UtcNow.AddDays(7), // תוקף פג SigningCredentials = creds }; // יצירת הטוקן ומשלוח כטקסט החוצה var tokenHandler = new JwtSecurityTokenHandler(); var token = tokenHandler.CreateToken(tokenDescriptor); return tokenHandler.WriteToken(token); }
